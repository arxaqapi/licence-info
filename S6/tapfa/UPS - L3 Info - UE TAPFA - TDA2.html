Remarque sur la fermeture :
lors de la définition d’une fermeture, OCaml capture les variables libres.

Une variable libre est une variable qui n’est pas liée par un fun var -> ... ou let var = ... in
OCaml va chercher la définition des variables libres dans l’environnement (si elles ne sont pas définies, on obtiendrait une erreur de syntaxe “Unbound value…”)

cf. la question 1 de la feuille de TD1.

dans let g = fun x -> sin x +. x,

  • x est liée
  • sin est libre

Polymorphisme et ηη-reduction

Une fonction polymorphe est… une fonction dont le type comporte des variables de type ('a, 'b, etc…) (pouvant donc prendre en argument des expressions de “n’importe quel type”)

On parle de polymorphisme paramétrique en OCaml, tandis que dans d’autres langages comme le C++, il y a juste du polymorphisme ad-hoc (surcharge d’opérateurs).

Le polymorphisme faible survient lorsqu’on effectue l’application partielle d’une fonction polymorphe. On peut le contourner en effectuant une ηη-reduction (c’est-à-dire en ajoutant une abstraction suivie immédiatement d’une application : fun x -> application partielle x )

Remarque sur la notion de Call-by-value

https://en.wikipedia.org/wiki/Evaluation_strategy#Call_by_value

OCaml suit une stratégie par Call by Value, càd que pour tous les appels de paramètres effectifs sont évalués d’abord, et la valeur correspondante est passée au paramètre formel de la fonction.










let rec f2 n = if n <= 0 then 0 else 1 + f2 (n - 1)
(* non récursif terminal *)

let rec f3 n = if n <= 0 then 0 else f3 (1 + f3 n)                  
(* non récursif terminal *)

let rec f4 r n = if n <= 0 then r else f4 (r + 1) (n - 1)
(* récursif terminal *)
let ok = f4 0 1_000_000;;

Questions diverses

  • à quoi sert la récursivité terminale ?

    1. ne pas forcément dire “à quoi sert la récursivité terminale”, puisque au départ c’est juste une propriété de certaines fonctions récursives ;
      mais dire à quoi sert l’optimisation des appels récursifs terminaux ?
      pour éviter d’empiler l’adresse de retour dans la pile d’appel et avoir un calcul récursif pour ces fonctions qui le satisfont, aussi efficace qu’une boucle imperative.
    2. à quoi ça sert de transformer une fonction non récursive terminale en fonction récursive terminale ?
      bénéficier de cette optimisation
    3. faut-il toujours faire cette transformation ?
      non : ne pas le faire systématiquement :
      • le faire pour éviter Stack Overflow sur de très longs calculs récursifs ;
      • dans le cas général, le faire si ça n’ajoute aucun surcout en lisibilité (par ex., fold_left plutôt que fold_right)

  • pourquoi on fait toujours 2 fun pour fold left?

    parce que le type de fold_left est ('r -> 'e -> 'r) -> 'r -> 'e list -> 'r,
    donc la fonction passée en argument attend bien deux arguments (res : 'r, et elt : 'e),
    d’où les deux fun (fold_left (fun res -> fun elt -> ...) ...)
